# 自动更新（GitHub Releases）

本项目内置「应用内检查 + 一键更新」能力。用户在 API 设置页（或画布右下角的提示）即可看到新版本并一键升级，**所有本地用户数据都会被保留**。

---

## 一、首次配置（维护者只做一次）

更新功能需要知道去哪个 GitHub 仓库查版本。打开 `main.py`，找到：

```python
APP_VERSION = "1.0.0"
GITHUB_REPO = os.getenv("MOISTCANVAS_REPO", "your-github-username/MoistCanvas").strip().strip("/")
```

把 `your-github-username/MoistCanvas` 改成你真实的仓库，例如 `zhangsan/MoistCanvas`。
（也可以不改代码，改成在运行环境里设置环境变量 `MOISTCANVAS_REPO=zhangsan/MoistCanvas`。）

未配置时，更新按钮会显示「未配置 GitHub 仓库」并禁用，不会报错。

> 可选：私有仓库或想提高 GitHub API 限额时，设置环境变量 `GITHUB_TOKEN=<你的 token>`。公开仓库不需要。

---

## 二、发布一个新版本（每次发版）

1. **改版本号**：把 `main.py` 里的 `APP_VERSION` 往上调，例如 `"1.0.0"` → `"1.1.0"`。
   版本号是判断「有没有新版」的唯一依据，必须每次发版都加。
2. **提交并推送**到 GitHub。
3. **在 GitHub 上创建 Release**：
   - tag 填 `v1.1.0`（前面的 `v` 可有可无，`1.1.0` 也行）。
   - Release 标题和正文随意填；**正文（body）会原样显示给用户当更新日志**，建议写清楚改了什么。
   - 发布（Publish release）。

完成后，所有运行旧版本的用户在下次打开应用 / 点「检查更新」时，就会看到 `v1.1.0`。

### 版本号比较规则

- 按数字分段比较：`1.0.10` > `1.0.9`（不是按字符串）。
- `v` 前缀会被忽略：`v1.1.0` 等价 `1.1.0`。
- 预发布后缀会被忽略：`1.2.0-beta` 视为 `1.2.0`。
- 只有「线上 tag 严格大于本地 APP_VERSION」时才提示更新。

---

## 三、更新时会动哪些文件

更新会下载该 Release 的源码包，只覆盖**代码**，**绝不碰用户数据**。

| 会被更新（代码） | 永远保留（用户数据） |
| --- | --- |
| `main.py` | `API/.env`（API 密钥） |
| `static/`（全部前端） | `output/`（生成的图片 / 视频） |
| `requirements.txt` | `data/canvases_v2/`（画布） |
| `README*`、`scripts/`、`.bat` | `data/*_cache.json`、`data/api_providers.json` |
|  | `history.json` |
|  | `runtime/`（便携 Python，不会重新下载） |

保护逻辑见 `main.py` 的 `_skip_path_for_update()`。另外，因为 `.gitignore` 已经排除了所有用户数据，GitHub 源码包里本身就不包含这些文件，等于双重保险。

### 依赖处理（先装依赖，再换代码）

更新时会对比 `requirements.txt` 是否变化：

- **没变** → 不动依赖，秒级完成。
- **变了** → 先用便携 Python 跑 `pip install -r requirements.txt`（针对下载下来的新依赖清单）。
  - **安装成功** → 才覆盖代码，再提示重启。
  - **安装失败** → **整次更新取消，代码一个字节都不动**，应用仍停在可正常运行的旧版本，界面提示「依赖安装失败……」。

> 这一顺序很关键：绝不允许出现「代码已是新版、依赖还是旧的」这种重启即崩的中间态。

### 关于 `data/api_providers.json`（重要约定）

整个 `data/` 目录都被当作用户数据保护，**不会被更新覆盖**——否则会清掉用户自己配置的平台 / 模型。
因此：**默认平台必须写在代码里的 `default_api_providers()`，不能只改 `data/api_providers.json`。**
应用加载时 `merge_default_api_providers()` 会把代码里的默认平台合并进用户配置，所以「在代码里加默认平台」既能随更新下发，又不会覆盖用户数据。如果只改 JSON 不改代码，老用户拿不到这个变更。

---

## 四、用户视角的更新流程

1. 打开应用，右下角出现「发现新版本 vX」提示（或进入 API 设置页顶部的更新条）。
2. 在 API 设置页点「一键更新」，等待下载 + 应用完成。
3. 点「立即重启」，应用会自动重启并刷新页面（仅 Windows）。
   - 若自动重启失败，关闭运行窗口、重新双击 `运行文件.bat` 即可。

`main.py` 被新版替换后，新文件自带新的 `APP_VERSION`，重启后版本号自动变新——无需额外维护版本文件。

---

## 五、安全与健壮性

- **只能本机触发更新 / 重启**：`/api/apply-update` 和 `/api/restart-app` 会校验**真实 TCP 连接来源**（`request.client.host`）必须是回环地址。因为服务绑定 `0.0.0.0`，这能挡住局域网机器（即使它伪造 `Host: localhost:3000` 也没用，`Host` 头是可伪造的，连接来源 IP 不可伪造）。代价是无法从「另一台机器的浏览器」触发更新——这是有意为之，更新会替换代码并重启服务，必须严格限制在本机。
- **防 CSRF**：本机浏览器里的恶意网页虽然来源也是回环，但会被同源校验（`Origin` 必须等于 `Host`）拦下，返回 403。
- **下载 / 解压有上限**：下载超过 200MB、解压总体积超 500MB、单文件超 80MB、文件数超 2 万，或包含 `..` / 绝对路径（zip-slip）都会被拒绝。
- **覆盖代码原子化 + 可回滚**：每个文件先写入同目录的临时文件，再用 `os.replace()` 原子替换——所以目标文件要么是完整旧版、要么是完整新版，**绝不会出现写一半被截断**的情况（即使磁盘在写入途中写满）。替换前还会把旧文件备份到 `runtime/_update/backup`；若某个文件失败，已替换的文件会从备份还原、新建的半成品会被删除，再报错——不会留下「半新半旧」的安装。
- **依赖先装后换码**：见上文「依赖处理」，pip 失败则整次取消、不动代码。

## 六、相关接口（供排查）

| 接口 | 说明 |
| --- | --- |
| `GET /api/app-version` | 返回当前版本、仓库、是否已配置 |
| `GET /api/check-update` | 查询 GitHub 最新 Release 并比较版本 |
| `POST /api/apply-update` | 下载并应用更新（仅本机；保留用户数据；先装依赖再覆盖；可回滚） |
| `POST /api/restart-app` | 重启服务（仅本机；仅 Windows） |
